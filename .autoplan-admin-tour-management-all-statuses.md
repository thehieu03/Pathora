## PHASE 1: CEO REVIEW (SELECTIVE EXPANSION MODE)

### Step 0A: Premise Challenge

**Premise 1 (hard-coded filter):** VALID. Source confirms `TourRepository.FindAll()` hard-codes `Status == TourStatus.Active` at line 92. Admin can't see non-active tours through the admin list.

**Premise 2 (client-side filtering is broken):** VALID and more serious than stated. The current architecture fetches 10 items per page from a dataset that only contains Active tours, then filters those 10 items client-side. This means:
- "pending" filter: never shows any results (backend only returns Active)
- "inactive" filter: never shows any results
- "rejected" filter: never shows any results
- Even "active" filter is misleading: it shows the first 10 Active tours, not all active tours
The plan correctly identifies the root cause but understates the severity.

**Premise 3 (client-side stats are wrong):** VALID. `statCounts` at line 228-233 computes counts from the 10-item page, not from the total. The `draft` count at line 232 always returns 0 (backend never returns draft status). These stats are decorative noise.

**Premise 4 (draft filter option):** VALID. "draft" doesn't exist in the backend enum (Active/Inactive/Pending/Rejected). The option at line 332 always returns empty results.

**Premise 5 (rejected option missing):** VALID. The backend has `Rejected` status but the dropdown at line 325-334 doesn't include it.
