## PHASE 3: ENG REVIEW

### Step 0: Scope Challenge

### Architecture Review

**CRITICAL FINDING #1: Backend requires all scalar fields in FormData**

The plan proposes "minimal FormData with only status field" (task 3.3). **This will fail.**

Backend investigation confirmed:
- `TourController.Update()` at line 184 uses `[FromForm] string tourName, [FromForm] string shortDescription, [FromForm] string longDescription` — all non-nullable `string`, no defaults
- `UpdateTourCommandValidator` marks these three as `NotEmpty()`
- Sending FormData without `tourName`/`shortDescription`/`longDescription` = HTTP 400 with validation errors

**Fix (required):** `handleStatusChange` must build FormData with ALL required scalar fields from the loaded `TourDto`: `id` + `tourName` (from `tour.tourName`) + `shortDescription` + `longDescription` + `status`. The `updateTourStatus` helper should accept the full `TourDto` as input to derive these.

**CRITICAL FINDING #2: Race condition — concurrent edits clobber each other**

When `handleStatusChange` captures `tour.tourName` etc. from the loaded page state, any concurrent edit (another tab, another admin) will be silently overwritten. The backend `UpdateTourCommand` applies all fields, not just status.

**Fix:** Document as known limitation (not fixable without backend change). Recommend admins avoid editing the same tour from multiple sessions. The proper fix is a dedicated PATCH endpoint with optimistic locking.
