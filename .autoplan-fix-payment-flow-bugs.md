## Corrected Fix Descriptions (post-review)

### Fix 1 (CORRECTED): Persist Booking Status on Payment Completion

**Root cause:** `BookingRepository.GetByIdAsync()` uses `AsNoTracking()`. Entity not tracked → `MarkDeposited/MarkPaid` changes invisible to EF Core → `SaveChangesAsync()` persists nothing.

**Call sites needing fix (2 total):**
1. `PaymentService.UpdateBookingStatusAsync()` — calls `booking.MarkDeposited/MarkPaid` directly
2. `PaymentReconciliationService.ReconcileProviderCallbackAsync()` — calls `ProcessPaymentCallbackAsync` which calls `MarkDeposited/MarkPaid` via the same AsNoTracking entity

**Solution:** Add `UpdateAsync(BookingEntity booking)` to `IBookingRepository` + `BookingRepository`. Call it after `MarkDeposited/MarkPaid` in BOTH call sites. Wrap in try-catch for graceful degradation.

### Fix 2 (VERIFIED): PayOS Webhook Date Parsing

**Status:** UNVERIFIED — need to read `PayOSWebhookPayload.cs` and compare against actual PayOS API response format. Don't code until verified.

### Fix 3 (CONFIRMED CRITICAL): Sepay Webhook HMAC Verification

**Root cause:** No signature verification on Sepay webhook endpoint. Any actor can POST fake payment notifications.

**Solution:** Add `ISepaySignatureVerifier` with HMAC-SHA256. Return 403 on invalid signature. Add to DI. Deploy ASAP.

### Fix 4 (CORRECTED): Normalize Sepay Amount Matching

**Diagnosis correction:** `PaymentReconciliationService.FetchMatchingProviderTransactionAsync` already uses `amount_in ?? amount_out` at line 280. **Only the outbox worker** (`OutboxWorkerService.CheckPaymentFromSePayAsync` at line 209) uses `amount_in` only.

**Also:** `amount >= expectedAmount` at line 280 should be `amount == expectedAmount` for security. Larger amounts should NOT match.
